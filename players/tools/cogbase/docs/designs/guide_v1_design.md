# guide_v1: Generation Pipeline Design

## Overview

guide_v1 generates a suite of 14 structured reference documents for a game,
enabling AI agent development from zero familiarity to working implementation.
The pipeline analyzes game source code through multiple LLM runners, synthesizes
their outputs, and produces documents in dependency order so that later documents
can build on earlier ones.

The design target is general-purpose for Coworld games and similar
inspectable agent/game codebases. Among Them is a validation target, not the
scope boundary.

`guide_v1` is a meta-pipeline toolkit. The guide documents and
`guide_contract.json` files it creates are generated artifacts and should live
under `output/<game_slug>/`, separate from the generator code and prompt
templates.

`guide_v1` is also the canonical front door for deciding what kind of
downstream artifacts a game needs. It should classify the player observation
surface as symbolic, visual, or mixed before any perception parser, capture
tool, or explorer artifact is generated. The deprecated `eyes_v1` prototype can
still inform targeted visual-artifact work, but it should not compete with
`guide_v1` as a source of project truth.

Guide generation also points runner prompts at this repository's Cyborg policy
framework, `agent_policies.frameworks.coborg`. The framework does not override
game-source facts, but it is the target architecture for implementation guidance:
percepts update belief,
deterministic modes emit symbolic intents, action resolution lowers intents to
transport actions, and slower strategies emit validated mode directives.

The canonical downstream consumer is `maker_v2`, a fresh scaffold that is
intended to replace the deprecated `maker_v1`. `maker_v2` is contract-first
and is meant to lean on agent-driven generation rather than hand-coded Python
extraction; it is not yet implemented beyond the CLI scaffold. See
`maker_v2_design.md`. While `maker_v2` is being built up, the deprecated
`maker_v1` (see `maker_v1_deprecation.md` and `maker_v1_design.md`) remains
runnable for short-term continuity: it reads `guide_contract.json` first,
falls back to Markdown for older bundles, writes build-plan artifacts,
generates starter symbolic agents for symbolic-primary games, emits live
visual starter agents when the guide proves a serializable action wire
contract, and can run offline visual bootstrap over captured image frames
with mock labels or AWS Bedrock Claude labels.

## Architecture

### Pipeline Shape

```
Game Source (local directory; URL support is a future extension)
    │
    ▼
┌─────────────────────────────────────┐
│         Stage-Based Generator        │
│                                      │
│  For each stage (1-7):               │
│    For each document in stage:       │
│      ├─ Runner A (Codex CLI)  ──┐    │
│      └─ Runner B (Claude CLI) ──┤    │
│                                 ▼    │
│      Synthesizer (Claude CLI)        │
│             │                        │
│             ▼                        │
│      Final document (.md)            │
│                                      │
│  Documents from earlier stages are   │
│  available as context for later ones │
└─────────────────────────────────────┘
    │
    ▼
Contract Builder (guide_contract.json)
    │
    ▼
Output Directory (generated guide artifact bundle)
```

### Runner Model

Each document is generated by:

1. **One or two independent runners** (Claude CLI and/or Codex CLI) that explore
   the game source and produce a draft of the document. The default remains
   Claude + Codex.
2. **One synthesizer** (Claude CLI) that takes both drafts and produces the
   final document when two runners are selected. If exactly one runner is
   selected, that runner's draft is promoted directly to the final document and
   synthesis is skipped.

Documents within a stage can run concurrently. Within one document generation,
the current implementation runs selected runners in the requested order, then
runs synthesis only when two runner drafts are available.

## Generation Order

Documents are generated in dependency order. Within each stage, all documents
can be generated concurrently.

### Stage 1: Foundation

| Document | Depends On |
|----------|-----------|
| `GAME_OVERVIEW.md` | — (only game source) |

### Stage 2: Pillars

| Document | Depends On |
|----------|-----------|
| `RULES_AND_MECHANICS.md` | GAME_OVERVIEW |
| `INTERFACE_CONTRACT.md` (agent-game protocol: observations, actions, rewards, timing) | GAME_OVERVIEW |

### Stage 3: Structure

| Document | Depends On |
|----------|-----------|
| `STATE_AND_VIEW_MODEL.md` | RULES_AND_MECHANICS |
| `CONNECTION_AND_EPISODE_LIFECYCLE.md` | INTERFACE_CONTRACT |
| `TRAINING_AND_EVALUATION.md` | INTERFACE_CONTRACT |

### Stage 4: Details

| Document | Depends On |
|----------|-----------|
| `OBSERVATION_DECODING.md` | INTERFACE_CONTRACT, STATE_AND_VIEW_MODEL |
| `ACTION_SEMANTICS_AND_CONTROL.md` | INTERFACE_CONTRACT, STATE_AND_VIEW_MODEL |
| `MEMORY_AND_HIDDEN_INFORMATION.md` | STATE_AND_VIEW_MODEL |
| `REWARDS_AND_PROGRESS_SIGNALS.md` | STATE_AND_VIEW_MODEL |
| `MINIMUM_VIABLE_AGENT.md` | CONNECTION_AND_EPISODE_LIFECYCLE |

### Stage 5: Integration

| Document | Depends On |
|----------|-----------|
| `ERROR_RECOVERY_AND_ROBUSTNESS.md` | INTERFACE_CONTRACT, STATE_AND_VIEW_MODEL |
| `STRATEGY_AND_POLICY_GUIDE.md` | RULES_AND_MECHANICS, STATE_AND_VIEW_MODEL |

### Stage 6: Internals

| Document | Depends On |
|----------|-----------|
| `IMPLEMENTATION_NOTES.md` | All prior documents |

### Stage 7: Index

| Document | Depends On |
|----------|-----------|
| `README.md` | All prior documents |

## Runner Invocation

Runners are invoked as CLI subprocesses:

- **Codex CLI**: `codex` (OpenAI's coding agent)
- **Claude CLI**: `claude` (Anthropic's Claude Code)

Both are given:
- A prompt (via stdin or `-p` flag)
- Access to the game source directory
- Access to the in-repo `agent_policies.frameworks.coborg` framework directory
  and its Python source root
- An output location to write their draft

The runners use their built-in tool use (file reading, web search, etc.) to
explore the game source. We do not pre-digest the source for them.

## Prompt Structure

Each runner receives a compact prompt with:

1. **Role**: You are generating documentation for an AI game agent.
2. **Game source location**: Path to explore.
3. **Agent framework location**: Path and package name for the Cyborg framework
   that downstream Maker artifacts should use.
4. **Document spec**: What this specific document covers (jurisdiction, scope).
5. **Prior documents**: List of already-generated docs with file paths and
   one-line descriptions. The runner is instructed to read these for context.
6. **Output format**: Markdown, written to a specified path.

The prompt is intentionally short. Runners are expected to use their tool-use
capabilities to read the source and prior docs themselves. We provide pointers,
not content.

### Example Prompt Shape

```
You are generating a reference document for AI agent developers working on
the target game.

## Your Task

Generate: STATE_AND_VIEW_MODEL.md
Purpose: Complete state/view graph with transitions, game phases, and how
they connect.

## Game Source

Explore the game source at: /path/to/game/source/
Use your file-reading tools to understand the game's implementation.

## Prior Documents (read these for context)

- /output/GAME_OVERVIEW.md — What the game is, entities, vocabulary
- /output/RULES_AND_MECHANICS.md — Logical game rules, win conditions

## Scope and Jurisdiction

This document OWNS:
- The complete state graph (all states and transitions)
- Game phases and their boundaries
- View model (what information is available in each state)

This document does NOT cover:
- How observations are encoded (that's OBSERVATION_DECODING)
- Action effects or timing (that's ACTION_SEMANTICS_AND_CONTROL)
- Connection/protocol details (that's CONNECTION_AND_EPISODE_LIFECYCLE)

## Output

Write the document to: /output/STATE_AND_VIEW_MODEL.md
```

## Synthesis Step

After both runners produce drafts, the synthesizer (Claude CLI) receives:

1. Both draft documents
2. The same document spec (jurisdiction, scope)
3. Instructions to merge, resolve conflicts, and produce a single coherent doc

The synthesizer prompt emphasizes:
- Prefer specificity over generality (concrete details from source > vague summaries)
- Resolve contradictions by referencing the source
- Maintain jurisdiction boundaries (don't let scope creep in)
- Produce a clean, final document (not a merge artifact)

## CLI Interface

```bash
# Generate all guides for a game
python generate_guides.py /path/to/game/source --output-dir ./output/my_game

# Generate specific documents
python generate_guides.py /path/to/game/source --only state-and-view-model

# Generate up to a specific stage
python generate_guides.py /path/to/game/source --through-stage 3

# Dry run (show what would be generated and in what order)
python generate_guides.py /path/to/game/source --dry-run

# Reuse existing documents and limit stage concurrency
python generate_guides.py /path/to/game/source --skip-existing --max-parallel 2

# Override runner models
python generate_guides.py /path/to/game/source --claude-model sonnet --codex-model gpt-5.5

# Select runner CLIs; one selected runner disables synthesis
python generate_guides.py /path/to/game/source --runner clod
python generate_guides.py /path/to/game/source --runner claude --runner codex

# Explicitly override the Cyborg framework for a compatibility experiment
python generate_guides.py /path/to/game/source \
  --agent-framework-dir /path/to/compatible/coborg
```

When no framework override is provided, Guide uses
`src/agent_policies/frameworks/coborg` from this repository. It does not
search external framework checkouts.

## Machine-Readable Contract

Every non-dry-run generation that produces at least one guide document writes
`guide_contract.json` beside the Markdown. The current schema version is
`guide.contract.v1`.

The contract is the downstream machine API for Maker. It records:

- generated time, game slug, source path, guide bundle hash, present documents,
  missing core documents, and missing full-suite documents;
- observation surface category, confidence, scores, primary channel,
  transport, input kind, encoding, dimensions, byte length, bit depth, and
  evidence lines;
- action style, default action, message-type requirement, payload prefix,
  action payloads, action candidates, and evidence lines;
- runtime endpoints, tick rate when discovered, and lifecycle notes.

The Markdown documents remain the human reference and source of richer context.
`guide_contract.json` is the stable handoff so later stages do not need to
depend on exact Markdown wording or table layout.

## Design Decisions

### Why CLI subprocesses instead of direct API calls?

The Claude and Codex CLIs come with their own harnessing: memory, tool use,
file access, web search, context management, skills, and settings. We get all
of that for free by invoking them as CLIs. We are orchestrating agents, not
building an LLM wrapper.

### Why default to two runners + synthesizer?

Different runners have different strengths in code exploration. Running two
independently and synthesizing produces more comprehensive coverage than a
single pass. The cost is manageable for final output (3 LLM calls per document,
42 total for the full suite), while `--runner` allows cheaper single-runner
debugging runs that skip synthesis.

### Why dependency-ordered stages?

Later documents benefit enormously from having earlier documents as grounding.
A doc about observation decoding is much better when it can reference an
already-established interface contract and state model. Sequential stages with
pointers to prior docs give later runners a strong foundation without us
needing to stuff everything into a single massive prompt.

### Why pointers instead of inlining prior docs?

Inlining all prior documents into later prompts would quickly exhaust context
windows (by Stage 5-6 we'd have thousands of lines of prior docs). Instead,
we give runners file paths and instruct them to read what they need. This lets
the runner's built-in context management handle the budget.

### Visual And Symbolic Surface Classification

Every generated guide bundle should make the player observation surface
explicit:

- **Symbolic primary**: the agent receives structured JSON, token arrays, game
  state, or another non-pixel contract.
- **Visual primary**: the agent receives pixels or render packets and must
  recover game state through perception.
- **Mixed/alternate**: the primary player contract is visual but source exposes
  a supported structured player path, native observation array, replay/global
  data, or other side channel.

This classification belongs in `INTERFACE_CONTRACT.md` and is interpreted in
`OBSERVATION_DECODING.md`. It drives downstream artifact decisions: symbolic
games may need decoders and policy scaffolds; visual games may need frame
fixtures, pixel/sprite parsers, source instrumentation, or manually seeded
capture harnesses.

## Resolved Decisions

1. **Error handling**: Retry once on failure, then skip the failed document and
   log a warning. If a failed document has downstream dependents in later stages,
   those dependents are also skipped (they'd lack their foundation). A Stage 4
   failure is isolated; a Stage 1 failure cascades and aborts the pipeline.

2. **Incremental regeneration**: Default is overwrite. If you re-run, you get
   fresh output. `--skip-existing` is implemented for continuing partially
   completed runs or preserving existing generated documents. Dependency-aware
   selective regeneration remains a v2 concern.

3. **Validation pass**: There is no dedicated validator yet. The synthesis
   prompt asks the synthesizer to resolve contradictions against source, but
   systematic validation should be designed from observed failure modes.

## Validation Targets

The pipeline has been exercised on multiple games, including **Among Them**,
**Cogs vs Clips**, and **Paint Arena**. Those runs are useful regression
examples because they cover different protocol and observation styles.
