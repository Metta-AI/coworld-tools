## Global shape

```
cogames [--install-completion] [--show-completion] [-h/--help] <command> ...
```

Top-level command groups (as displayed by `cogames --help`):

| Group | Commands |
|---|---|
| Missions | `missions`, `evals`, `variants`, `describe`, `make-mission` |
| Play | `play`, `replay` |
| Evaluate | `scrimmage`, `run`, `pickup`, `diagnose` |
| Info | `version`, `docs` |
| Policies | `policies`, `create-bundle`, `validate-bundle` |
| Tournament | `submissions`, `leaderboard`, `matches`, `match-artifacts`, `upload`, `submit`, `ship`, `auth`, `season`, `episode`, `assay` |
| Tutorials | `tutorial` |
| BitWorld | `bitworld` |

Conventions observed across commands:

- `-h` / `--help` is universal.
- `-p` is always **policy**.
- `-m` is always **mission**.
- `-v` is always **variant** (usually repeatable).
- `-c` is always **cogs** (agent count).
- `-s` has **two meanings** depending on command: "steps" in simulation
  commands (`play`, `scrimmage`, `run`, `pickup`, `diagnose`), and
  "server URL" / "season" in tournament commands. The CLI explicitly warns
  "use `--seed`, not `-s`" in simulation commands.
- `-n` is "name" (for uploads) or "limit" (for list commands). Context-dependent.
- `-f` is "include-file" in bundling contexts.
- `-o` is always "output file/dir".
- `-k` is "init kwarg".
- Output flag `--json` is available on almost every data-returning command.

## Known quirks

1. **`cogames missions` is flaky.** The positional `MISSION` argument is
   odd — `cogames missions` (no args) falls through to an empty-filter
   listing that prints `No missions matching 'missions'`. Passing any
   non-subcommand-lookalike string (`arena`, `machina`, `.`, `a`) raises a
   typer `UsageError`. Passing `missions` as the filter (i.e.
   `cogames missions missions`) returns the help page. The only way this
   subcommand seems to work in practice is `cogames missions <prefix>`
   where `<prefix>` happens to not collide with click's metadata arg
   parser. If you need a reliable mission list, use **`cogames describe
   <mission>`** against a known mission name, or open
   `~/coding/metta/packages/cogames/src/cogames/` source.
2. **`cogames auth login` is the right entry point**, not `softmax login`
   or `uv run softmax login` — those use a different package
   (`softmax-cli`) which may not be in your active env. The `cogames auth
   status` "Run softmax login first" message is misleading.
3. **`cogames bitworld *` requires the `bitworld` extra.** Without it
   you'll see: `'cogames bitworld' requires the optional BitWorld package.
   Install it with: pip install cogames[bitworld]`.
4. **Two kinds of login.** `auth login` is for your user session.
   `auth player` manages separate **player** sessions and per-player API
   credentials. `player login <PLAYER>` swaps the active session to
   short-lived player-scoped tokens; `player logout` restores the user
   session.
5. **`-s` ambiguity.** In `play`, `scrimmage`, `run`, `pickup`, `diagnose`,
   `-s` is `--steps` (per-episode limit). In `leaderboard`, `matches`,
   `upload`, etc. `-s` is `--server` (tournament URL). Help text calls
   this out.
6. **`season matches` vs. top-level `matches`.** `cogames matches` is
   your matches (filtered by policy). `cogames season matches <SEASON>`
   is *all* matches in a season.

## Shared option patterns

### Server/auth (tournament commands)

Almost every tournament command accepts:

```
--server / -s URL        Tournament server URL
                         [default: https://api.observatory.softmax-research.net]
--login-server URL       Authentication server URL
                         [default: https://softmax.com/api]
--json                   Print raw JSON instead of table
```

### Policy specification

A `POLICY` argument (passed with `-p` / `--policy`) accepts three forms:

1. **URI (checkpoint bundle)** — points at a directory or `.zip` that
   contains `policy_spec.json`:
   - `./train_dir/my_run:v5`
   - `./train_dir/my_run:v5.zip`
   - `s3://bucket/path/run:v5.zip`
   - `./train_dir/checkpoints:latest` (auto-resolves highest version)
2. **Shorthand name** — from `cogames policies`:
   - `class=random`, `class=baseline`, `class=starter`, `class=lstm`, etc.
3. **Explicit class+data+kwargs**:
   - `class=<path.to.Class>[,data=<weights>][,proportion=<float>][,kw.x=y]`
   - Example: `class=lstm,data=./weights.safetensors,proportion=3`

For evaluation commands supporting mixed policies, `proportion=N` sets
relative agent allocation (e.g. `-p foo,proportion=3 -p bar,proportion=5`
→ 3:5 mix).

### Mission specification

`-m` / `--mission` accepts:

- A name from the registry: `arena`, `machina_1`, `training_facility_1`,
  `four_score`, `hello_world`, etc.
- A path to a YAML/JSON mission config: `path/to/mission.yml`.
- Wildcards (in `run` / `scrimmage` / `tutorial train`): `'arena.*'`,
  `'*:shaped'`, `'machina*:shaped'`, `'machina_2_bigger:*'`.

### Variants

`-v` / `--variant` is repeatable and applies modifiers. Known variants
(from `cogames variants`) include: `miner_role`, `aligner_role`,
`scrambler_role`, `scout_role`, `team`, `talk`, `vibes`, `no_vibes`,
`damage`, `energy`, `solar`, `cargo_limit`, `territory`, `hearts`,
`junction`, `extractors`, `gear_stations`, `team_hub`, `multi_team`,
`forest`, `caves`, `desert`, `city`, `days`, `endless`, `small_50`,
`overrun`, and many more. See [`variants`](#variants) for the full table.

## Command reference

### Missions

#### `missions`

```
cogames missions [OPTIONS] MISSION

  List available missions.

Arguments:
  MISSION   Filter missions by name.

Options:
  --game TEXT         Game whose missions to list. [default: cogsguard]
  --dependencies      Show variant dependencies for each mission.
  --mission, -m M     Describe a specific mission (enables --cogs, --variant,
                      --format, --save flags).
  --cogs, -c INTEGER  Override agent count (requires -m).
  --variant, -v V     Apply variant (requires -m, repeatable).
  --format [yaml|json]  Output format (requires -m).
  --save, -s PATH     Save config to file (requires -m).
  --help, -h
```

Two modes: list all missions (no `-m`), or describe a specific mission
(with `-m`, at which point `--cogs` / `--variant` / `--format` / `--save`
apply).

> Quirk: the list mode currently misfires for almost all filter values.
> Use [`describe`](#describe) if you know the mission name. To enumerate
> missions, fall back to game docs or the cogames source.

#### `evals`


### Play

#### `play`

```
cogames play [OPTIONS]

  Play a game interactively. Single episode.

  Default policy is 'noop' (agents don't move). Use -p class=random or
  -p class=baseline to see autonomous movement. Multiple -p flags assign
  one policy per team.

  Manual control: in GUI, click a cog; in unicode mode, press M and use
  arrow/WASD keys. Log mode is non-interactive.

Game Setup:
  --game GAME              Game to play. [default: cogsguard]
  --mission, -m MISSION    Mission to play.
  --variant, -v VARIANT    Variant modifier (repeatable).
  --cogs, -c N             Number of cogs. [default: from mission]

Policy:
  --policy, -p POLICY      Policy per team (one -p all teams, multiple -p
                           assigns per team).
  --device DEVICE          auto|cpu|cuda|cuda:0|... [default: auto]

Simulation:
  --steps, -s N            Max steps per episode. [default: 1000]
  --action-timeout-ms MS   Max ms per action before noop. [default: 10000]
  --render, -r MODE        auto|gui|unicode|log|none [default: auto]
  --seed INTEGER           RNG seed. [default: 42]
                           (Use --seed, NOT -s, for seed.)
  --map-seed SEED          Separate seed for procedural maps.
                           [default: same as --seed]
  --autostart              Start without waiting for user input.

Output:
  --save-replay-dir DIR    Save replay for cogames replay.
  --save-replay-file FILE  Save replay to a fixed path (overwrites).
```

Examples:

```bash
cogames play -m arena
cogames play -m arena -p class=random
cogames play -m arena -c 4 -p class=baseline
cogames play -m four_score -p nlanky -p baseline -p random -p noop
cogames play -m four_score -p nlanky:1 -p random:2   # cycling mix
cogames play -m arena --save-replay-file ./latest.json.z
cogames play -m machina_1 -v talk -r gui             # speech bubbles
```

#### `replay`

```
cogames replay [OPTIONS] FILE

  Replay a saved game episode in the GUI.

Arguments:
  FILE                 MettaGrid replay (.json.z, .replay, .bin) or
                       BitWorld replay (.bitreplay) [required]

Options:
  --duration FLOAT     Seconds to keep a BitWorld replay server alive.
                       (MettaScope replays ignore this.)
```

Examples:

```bash
cogames replay ./replays/game.json.z          # CvC in MettaScope
cogames replay ./train_dir/my_run/replay.bin  # legacy MettaGrid
cogames replay ./among_them.bitreplay         # BitWorld global client
```

### Evaluate

#### `scrimmage`

```
cogames scrimmage [OPTIONS]

  Evaluate a single policy controlling all agents.
  (Equivalent to `cogames run` with one policy.)

Mission:
  --mission, -m MISSION   Missions to evaluate (supports wildcards).
  --cogs, -c N
  --variant, -v V

Policy:
  --policy, -p POLICY
  --device DEVICE          [default: auto]

Simulation:
  --episodes, -e N         [default: 10]
  --steps, -s N            [default: from mission]
  --seed N                 [default: 42]
  --map-seed N
  --action-timeout-ms MS   [default: 250]

Output:
  --format yaml|json
  --save-replay-dir DIR
```

Example: `cogames scrimmage -m arena -p lstm`.

#### `run`

```
cogames run [OPTIONS]

  Evaluate one or more policies on missions.

  With multiple policies, agents are split between them; which agents
  swap policy each episode. With one policy, equivalent to scrimmage.
```

Same flags as `scrimmage`. Supports `-p` repeated with `proportion=` for
mixed pools.

Examples:

```bash
cogames run -m arena -p lstm
cogames run -m machina_1 -p ./train_dir/my_run:v5
cogames run -m 'arena.*' -p lstm -p random -e 20
cogames run -m machina_1 \
    -p ./train_dir/my_run:v5,proportion=3 \
    -p class=random,proportion=5
```

#### `pickup`

```
cogames pickup [OPTIONS]

  Evaluate a policy against a pool of other policies and compute VOR
  (Value over Replacement).

Mission:
  --mission, -m MISSION   [default: arena]
  --cogs, -c N [x>=1]     [default: 4]
  --variant, -v V

Policy:
  --policy, -p POLICY     Candidate policy to evaluate.
  --pool POLICY           Pool policy (repeatable).
  --device DEVICE         [default: auto]

Simulation:
  --episodes, -e N        [default: 1]
  --steps, -s N           [default: 1000]
  --seed N                [default: 50]
  --map-seed N
  --action-timeout-ms MS  [default: 250]

Output:
  --save-replay-dir DIR
```

Example: `cogames pickup -p greedy --pool random`.

Use this *after* `play` and `scrimmage`, not as the first command.

#### `diagnose`

```
cogames diagnose [OPTIONS] POLICY

  Run diagnostic evals for a policy checkpoint.

Arguments:
  POLICY                         Policy spec. [required]

Evaluation:
  --mission-set, -S SET          [default: cvc_evals]
  --experiments NAME             Subset of mission set.
  --cogs, -c N                   Agent counts to test (repeatable).
  --device DEVICE                [default: auto]
  --scripted-baseline-policy POLICY  Reference scripted baseline.
  --known-strong-policy POLICY   Reference known-strong policy.

Simulation:
  --steps, -s N                  [default: 1000]
  --episodes, -e N               [default: 3]

Output:
  --output-dir DIR               Directory for diagnose artifacts.
  --compare-run-dir DIR          Previous diagnose run with doctor_note.json.
  --bundle-zip FILE              Write portable diagnose artifact zip.
  --stability-reruns N [x>=0]    Additional fixed-seed reruns. [default: 0]
  --require-stable-interpretation  Mark incomplete if stability fails.
```

Intended for deep policy checkups. Full Stage-2 diagnosis currently
requires `cvc_evals`.

#### `auth`

Subcommand group for user authentication.

```
cogames auth login [--login-server URL] [--no-browser] [--force/-f]
cogames auth logout [--login-server URL]
cogames auth get-login-url [--login-server URL]
cogames auth status [--login-server URL] [--server/-s URL]
cogames auth get-token [--login-server URL]
cogames auth set-token [--login-server URL] TOKEN
cogames auth player ...
```

`auth player` is a nested group for **per-player session management**:

```
cogames auth player list [--server URL] [--json]
cogames auth player login PLAYER [--server URL] [--json]
cogames auth player logout [--server URL]
cogames auth player create --name NAME [--server URL] [--json]
cogames auth player credentials list PLAYER [--json]
cogames auth player credentials create PLAYER --name NAME [--scope SCOPE ...] [--json]
cogames auth player credentials revoke PLAYER CREDENTIAL_ID
```

Credentials support scopes: currently `write`. `create` returns the token
**only once** — save it.

Use cases:

- `auth login` (user session): normal human dev workflow.
- `auth player login <name>`: swap to a short-lived player-scoped token
  for machine agents or shared infra.
- `auth player credentials create`: mint a long-lived API credential for
  a player (useful for CI/CD that submits on behalf of a player identity).

#### `season`

```
cogames season list                    [--json]
cogames season show SEASON             [--json]
cogames season versions SEASON         [--json]
cogames season stages SEASON           [--json]
cogames season progress SEASON         [--json]
```

All commands take the shared `--server / -s` and `--login-server` flags.
`season leaderboard` can be further scoped by `--pool` and `--type` for
stage-specific views. `season matches` lists *all* matches in a season
(top-level `matches` lists *yours*).

#### `assay`

Assay runs are more involved evaluation pipelines (e.g. `cvc_evals`,
`role_specific_evals`).

```
cogames assay status POLICY_OR_RUN_ID   [--json]
cogames assay list                      [--policy/-p POLICY] [--json]
cogames assay results POLICY_OR_RUN_ID  [--json]
cogames assay submit POLICY
    [--source TEXT]         # mission set, default cvc_evals
    [--name/-n TEXT]        # dedupe label
    [--compat-version/-c]   # override compat version
    [--episodes/-e N]       # default 3
    [--max-steps N]         # default 10000
    [--watch/-w]            # poll until complete
    [--json]
```

`assay submit` triggers a server-side eval run against a mission set.
Combine with `assay status <run_id> --watch` to track.

### Tutorial

```
cogames tutorial play
cogames tutorial cvc
cogames tutorial make-policy (--trainable | --scripted | --amongthem) [-o FILE]
cogames tutorial train [OPTIONS]
```

- `tutorial play` — guided Cogs-vs-Clips tutorial (GUI).
- `tutorial cvc` — roles & territory control deep-dive (GUI).
- `tutorial make-policy` — generate a starter template. **Exactly one**
  type flag required:
  - `--trainable` — neural network policy.
  - `--scripted` — rule-based scripted policy.
  - `--amongthem` — BitWorld AmongThem scripted practice template.
  - `-o` / `--output` — output path (default `my_policy.py`).
- `tutorial train` — train a policy. Requires the `neural` extra.

`tutorial train` flags (abbreviated):

```
--mission, -m MISSION     Repeatable; wildcards supported.
--cogs, -c N              [default: from mission]
--variant, -v V           Repeatable.
--policy, -p POLICY       [default: class=lstm]
--steps N                 [default: 10000000000]
--minibatch-size N        [default: 4096]
--device DEVICE           auto|cpu|cuda|mps
--num-workers N           [default: CPU cores]
--parallel-envs N
--vector-batch-size N
--seed N                  [default: 42]
--map-seed N
--checkpoints DIR         [default: ./train_dir]
--log-outputs
```

Examples:

```bash
cogames tutorial make-policy --amongthem -o amongthem_policy.py
cogames tutorial train -m arena
cogames tutorial train -m arena -p class=baseline
cogames tutorial train -p ./train_dir/my_run:v5           # continue
cogames tutorial train -p class=lstm,data=./weights.safetensors
cogames tutorial train -m mission_1 -m mission_2          # curriculum
cogames tutorial train -m 'machina_2_bigger:*'            # wildcards
cogames tutorial train -m '*:shaped'
cogames tutorial train -m 'machina*:shaped'
```

### BitWorld

Optional package (`pip install cogames[bitworld]`).

```
cogames bitworld games
cogames bitworld quick-run GAME [PORT]
    [--players N] [--address HOST] [--save-replay PATH] [--nim PATH]
cogames bitworld replay REPLAY_PATH
    [--game GAME] [--port N] [--address HOST] [--browser-address HOST]

```

- `games` — list installed BitWorld games.
- `quick-run` — compile and run a BitWorld game with local player clients.
- `replay` — replay a `.bitreplay` in the global web viewer.

Currently requires the optional extra; without it you'll see:
> `'cogames bitworld' requires the optional BitWorld package. Install it with: pip install cogames[bitworld]`
